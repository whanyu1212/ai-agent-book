# 第8章 · Agent の自己進化

> 重みを変えずに成長する。3 つの学習パラダイム、経験からの学習、そして「ツールの利用者」から「ツールの創造者」への道のりを通じて、Agent を「賢い」段階から「熟練した」段階へと進化させる。

## 付随プロジェクト

| プロジェクト | 種類 | 説明 |
| --- | :--: | --- |
| [gaia-experience](gaia-experience/) | ✅ | AWorld フレームワークと GAIA ベンチマークに基づき、完全な「学習・適用」ループを実装する。Agent は成功したタスクの軌跡を構造化された経験へと自動的に要約し、新しいタスクでそれらを取得・適用することで自己進化を実現する。 |
| [browser-use-rpa](browser-use-rpa/) | ✅ | ブラウザ自動化のためのワークフロー記録システムを実装し、繰り返される操作シーケンスをパラメータ化されたツールへと自動的にカプセル化する。高価な LLM 推論から正確な自動実行へ切り替えることで、3〜5 倍の速度向上を実現する。 |
| [prompt-distillation](prompt-distillation/) | ✅ | 複雑なプロンプトの有効性をモデルパラメータへと蒸留し、推論時のプロンプト長を削減して、コンテキスト上の経験をパラメータ化された知識へと固定化する。 |
| [prompt-auto-optimization](prompt-auto-optimization/) | ✅ | 人間のフィードバックに基づく自動化されたシステムプロンプト学習。tau-bench スタイルの航空会社カスタマーサービスの「過剰転送」問題を例に、Coding Agent がシステムプロンプトファイルを読み取り、問題のあるルールを特定し、正確な修正を生成し、実際にプロンプトファイルを書き換える。次に変更を再評価し、「フィードバック → 書き換え → 検証」のループを形成する。 |
| [self-evolving-tools](self-evolving-tools/) | ✅ | Alita スタイルの「事前定義は最小限、自己進化は最大限」というアプローチ。Agent は事前構築されたドメイン固有のツールを持たず、5 つの汎用メタツールのみを持つ。実行できないタスクに遭遇すると、Web でオープンソースのライブラリ/API を検索し、ドキュメントを読み、サンドボックスでテストし、実行可能な解決策を新しいツールとしてカプセル化し、再利用のためにツールライブラリに保存する。そのプロセス全体を通じてハルシネーション制御を重視する。 |
| [self-evolution-eval](self-evolution-eval/) | ✅ | Agent の「自己進化」能力（自らツールを発見、作成、再利用する能力）を評価するために設計された専用のデータセットと検証手法。20 個のクロスドメインタスク（ツール名のヒントなし）+ 四層の階層的検証 Harness + 制御可能な参照 Agent。「結果が正しいかどうか」の確認を超えて、発見、作成、再利用の質を評価する。 |
## プロジェクトの種類

| アイコン | 種類 | 意味 |
| :--: | --- | --- |
| ✅ | **単独実行** | このリポジトリに完全なコードがあり、API キーを設定すれば実行できる |
| 📖 | **再現ガイド** | `git clone` が必要な**外部リポジトリ**に依存する詳細ドキュメント |
| 🚧 | **設計ドキュメント** | アーキテクチャ/実装計画のみで、実行可能なコードは未完成 |
