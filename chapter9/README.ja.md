# 第9章 · マルチモーダルとリアルタイムインタラクション

> 知覚と行動をテキストから音声、GUI、そして物理世界へと拡張する。3 つの音声パラダイム（カスケード型/エンドツーエンドの全モーダル型/全二重型）、ストリーミング音声の知覚と合成、Computer Use、そしてロボット操作。

## 付随プロジェクト

| プロジェクト | 種類 | 説明 |
| --- | :--: | --- |
| [live-audio](live-audio/) | ✅ | 音声認識、AI 対話、音声合成を統合したリアルタイム音声チャットのデモ。複数の AI サービスプロバイダー（OpenAI、OpenRouter、ARK、Siliconflow）をサポートし、低レイテンシの対話体験を提供する。 |
| `browser-use/` | 📖 | Browser-Use は、LLM がブラウザを制御して複雑なタスクを完了できるようにする強力なブラウザ自動化フレームワークである。フォーム入力、Web ナビゲーション、データ抽出などのシナリオをサポートし、GUI 自動化（Computer Use）の典型的な実装として機能する。 |
| `claude-quickstarts/` | 📖 | Claude API のクイックスタート例とベストプラクティスで、さまざまなユースケースをカバーする。 |
| [phone-agent](phone-agent/) | ✅ | 「ユーザーに代わって電話で外の世界とやり取りする」音声 Agent を示す。上位層は標準的な ReAct Agent である。自然言語のタスクを受け取ると、電話の番号と目的を自律的に決定し、（電話 API 抽象化に基づく）`make_phone_call` ツールを呼び出して会話全体を完了し、構造化された通話ログを読み取り、必要に応じて再度電話をかけて追加の質問をし、最終的にユーザーに報告する。 |
| [end-to-end-speech](end-to-end-speech/) | ✅ | Step-Audio R1 のエンドツーエンド音声思考（「聞く → 考える → 話す」）に対応し、ASR→LLM→TTS のカスケードとレイテンシおよびパラ言語的損失を比較する。 |
| [streaming-speech](streaming-speech/) | ✅ | ストリーミング音声知覚の中核的なトレードオフを示す。連続した音声を徐々に長さを増すセグメントに分割して ASR に供給する。受信した各セグメントは「現在の部分的な認識結果」を生成し、早期のテキスト出力のために極めて低い最初のチャンクのレイテンシを実現する。その代償として、後半の文脈を欠く早期のチャンクは誤る可能性があるが、音声が蓄積するにつれて徐々に収束する。これは「文全体を待ってから認識する」高精度/高レイテンシのアプローチと対照的である。 |
| [controllable-tts](controllable-tts/) | ✅ | メイン LLM の出力には制御トークン（感情/話速/スタイル/ポーズ/笑い）が含まれる。実行層はこれらのトークンを解析し、参照音声ライブラリ内の対応するスタイルプロファイルにマッピングし、次に音声を合成する。これにより「どこでポーズを入れ、どんなトーンを使うか」の決定を LLM に委ね、同じテキストを異なるスタイルや感情で合成できるようにする。 |
## プロジェクトの種類

| アイコン | 種類 | 意味 |
| :--: | --- | --- |
| ✅ | **単独実行** | このリポジトリに完全なコードがあり、API キーを設定すれば実行できる |
| 📖 | **再現ガイド** | `git clone` が必要な**外部リポジトリ**に依存する詳細ドキュメント |
| 🚧 | **設計ドキュメント** | アーキテクチャ/実装計画のみで、実行可能なコードは未完成 |
