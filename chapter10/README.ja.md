# 第10章 · マルチ Agent の協調

> 集団的知性は個々の知性を超えうる。マルチ Agent の分類フレームワーク、それが本当に単一の Agent を上回るのはいつか、コンテキストを共有する協調と共有しない協調、失敗モード、そして創発する「Agent 社会」。

## 付随プロジェクト

| プロジェクト | 種類 | 説明 |
| --- | :--: | --- |
| `use-computer-while-calling/` | 📖 | Phone Call Agent と Computer Use Agent による二重 Agent 協調アーキテクチャを実装する。2 つの Agent はコーディネーターを介さず、WebSocket を通じて直接通信する。Phone Agent が音声インタラクションを担当し、Computer Agent がブラウザ自動化を実行し、音声と Web 操作の両方を必要とする複雑なタスクを並列で完了する。 |
| [staged-system-prompt](staged-system-prompt/) | ✅ | 同じ Coding Agent が、タスクの異なる実行段階（要件明確化 → コード実装 → コードレビュー）で異なるシステムプロンプトとツールセットを読み込む。これにより、単一の会話の中で異なる役割を演じ、異なる挙動を示すことができる。その間、対話履歴とタスク状態は各段階で継続的に共有される。レビューが失敗した場合は、実装段階へ戻ることができる。 |
| [multi-role-transfer](multi-role-transfer/) | ✅ | 共有コンテキスト下での連鎖的なハンドオフを示す。単一のセッションに複数の専門的な役割の Agent が含まれ、それぞれが独自のシステムプロンプトと専用のツールセットを持つ。`transfer_to_agent` ツールを用いて、Agent はタスクの進捗に基づいていつ別の役割に切り替えるかを自律的に決定する。同じ対話履歴を共有しているため、ハンドオフ時に完全なコンテキストが自然に保持される。 |
| [book-translation](book-translation/) | ✅ | オーケストレーターモードを用いて、長文ドキュメントの翻訳を用語集/翻訳/校正の専門 Agent へと分解する。Manager はタスク、計画、呼び出し記録、ファイルインデックスのみを保持し、完全な訳文はディスクに書き出されるため、コンテキストをほぼ一定に保つ。単一 Agent のアプローチと比較し、実際のトークン数を用いて、コンテキストの爆発をどう制御し、共有用語集によって書籍全体の一貫性をどう確保するかを示す。 |
| [parallel-web-research](parallel-web-research/) | ✅ | 中央調整を伴う複数の同質な Agent による並列検索を示す。メインコーディネーターが同時に N 個のサブ Agent を起動し、各 Agent が 1 つのソースにアクセスして答えを探す。1 つが目標に到達すると、他の Agent は正常に停止する。メッセージバス、並列ディスパッチ、リアルタイム監視、カスケード終了、競合状態の処理がすべて現実的に実装されている（実際のブラウザの代わりに制御可能なシミュレート情報源を使用）。 |
| [voice-werewolf](voice-werewolf/) | ✅ | マルチ Agent の人狼ゲームを用いて、「非共有コンテキスト」下での情報アクセス制御を示す。各プレイヤーは厳密に隔離されたプライベートコンテキストを持つ独立した LLM Agent である。コード駆動の決定論的な審判が、どの情報をどのプレイヤーのコンテキストに届けるかを決定し、監査のために登録し、ゲーム終了時に隔離の正しさを自動的に検証する。音声はオプションの拡張である。 |
## プロジェクトの種類

| アイコン | 種類 | 意味 |
| :--: | --- | --- |
| ✅ | **単独実行** | このリポジトリに完全なコードがあり、API キーを設定すれば実行できる |
| 📖 | **再現ガイド** | `git clone` が必要な**外部リポジトリ**に依存する詳細ドキュメント |
| 🚧 | **設計ドキュメント** | アーキテクチャ/実装計画のみで、実行可能なコードは未完成 |
